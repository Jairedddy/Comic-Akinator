Goal

One offline script that produces a canonical data/characters.json from public sources. Everything downstream — the engine, the UI, the sim — reads from this file.

Sources (all free, no key)

  • Kaggle Marvel + DC characters CSV — download once, save to scripts/sources/kaggle-marvel.csv and kaggle-dc.csv. Gives alignment, gender, eye/hair color, publisher, year of first appearance.
  • Superhero API JSON dump — pull from https://akabab.github.io/superhero-api/api/all.json. 731 characters with powerstats, biography, appearance, work, connections. Save to scripts/sources/superhero-api.json.
  • Wikipedia thumbnails — resolve image URLs by querying Wikipedia REST /page/summary/{name} (free, no key). Cache results to scripts/sources/image-cache.json so re-runs are fast.

Step-by-step

  1. Create scripts/sources/ directory and download the three sources once. Commit the CSV/JSON; do NOT commit image-cache.json initially — let it rebuild.
  2. Install pipeline-only deps: npm i -D csv-parse
  3. Build scripts/lib/normalize.ts with the canonical schema and helpers: slugify(name), dedupeKey(realName, publisher).
  4. Build scripts/build-dataset.ts doing in order: (a) load CSV + JSON sources → (b) normalize each into a partial Character → (c) merge on dedupeKey, preferring richer fields → (d) resolve image URLs via Wikipedia (with cache + 300ms rate-limit between requests) → (e) compute a traitCompleteness score per character → (f) select top 500 by completeness × popularity → (g) write to data/characters.json.
  5. Build scripts/validate-dataset.ts: assert every character has id, name, publisher, image, ≥ 8 non-null traits. Exit nonzero on violation (this gates npm run build).

M1.4Entropy engine module
Bayesian posterior + info-gain question selection + adaptive guess + wrong-guess loop.
▾

~10hcore IP

Goal

The brain. A pure-TypeScript module with no React, no DOM, no I/O. Inputs: dataset + current answer history. Outputs: next question or a guess.

Module layout (src/lib/engine/)

  • types.ts — Answer, Belief, GameState, EngineDecision
  • likelihood.ts — maps (answer, traitValue) → multiplier
  • posterior.ts — update(belief, question, answer, chars) with min-floor 1e-6, renormalize
  • entropy.ts — entropy(belief)
  • select.ts — nextQuestion(state, pool) with info-gain + diversity penalty (0.85x if category is in last 2)
  • guess.ts — shouldGuess(belief, turn) + shouldGiveUp(belief, turn, missCount)
  • index.ts — the high-level decide(state) orchestrator

Step-by-step

  1. Write types.ts first — pin the shapes before any logic.
  2. Implement likelihood.ts.
  3. Implement posterior.ts.
  4. Implement entropy.ts (Shannon, base 2).
  5. Implement select.ts with info-gain + diversity penalty.
  6. Implement guess.ts with the threshold table (turn ≥ 8 + top1 > 0.85 or top1/top2 > 6, soft cap 20, hard cap 25).
  7. Implement index.ts — returns { kind: 'question', q } | { kind: 'guess', characterId } | { kind: 'giveup' }.
  8. Build scripts/play-cli.ts: a tiny CLI that loops engine.decide() ↔ readline input, prints the chosen question, reads 1–5 from your terminal, and prints the engine’s guess at the end.

Manual Testing

The whole point of the CLI script is to exercise the engine by hand. Play a series of games against known characters; the engine should land the right guess inside 25 turns.

  1. Run npm run play. Pick Spider-Man in your head. Answer truthfully → engine guesses Spider-Man within 25 turns.
  2. Repeat for: Batman, Wonder Woman, Magneto, Thanos. All 5 must land the right guess.
  3. Play a 6th game as Spider-Man and answer "Don’t know" to half the questions — engine still narrows to a top-3 that includes Spider-Man.
  4. Play a 7th game and intentionally answer wrong for the first 3 questions, then truthfully — engine should recover and still land in ≤ 25 turns.
  5. Play a game as an obscure character not in the dataset → engine eventually emits kind: 'giveup' at or before turn 25.
  6. Inspect the CLI’s per-turn output: top-3 candidates + their posterior probabilities. Each Yes answer should visibly shift mass toward characters with that trait.
  7. Verify the engine has no window, localStorage, or React imports — grep src/lib/engine/ for those strings → zero hits.

Done when:

  • All 7 tests pass.
  • 5 of 5 known characters guessed inside 25 turns under truthful play.
  • Engine module is pure (no DOM/React/storage).
M1.5Simulation harness baseline

▾

~3himportant

Goal

An offline benchmark that plays a full game for every character in the dataset, using that character’s own traits as the oracle. Gives you a single number for "how good is the engine?" and a list of characters the engine struggles with.

Step-by-step

  1. Create scripts/sim.ts.
  2. For each character C: initialize a fresh GameState; loop calling engine.decide(state); answer using C’s trait value (1.0→Yes, 0.0→No, 0.4–0.6→Probably, null→Don’t know); record hit/miss; cap at 30 turns.
  3. Aggregate: % correctly guessed, average question count, P95 question count, top 20 characters by failure.
  4. CLI flag --noise=0.1 flips 10% of oracle answers to "Don’t know".
  5. Save the report to sim-report.json for diff-friendly tracking across commits.

Manual Testing

  1. Run npm run sim → finishes in under 60 s for 500 characters.
  2. Output prints summary: accuracy %, avg questions, P95 questions, list of top-20 failures.
  3. Open sim-report.json → one entry per character with { guessed, turns }.
  4. Run npm run sim -- --noise=0.1 → accuracy drops vs. baseline; report still completes.
  5. Open the top-3 failure characters in characters.json → eyeball why they failed (sparse traits? trait values that don’t differentiate them from siblings?).
  6. Manually corrupt a character’s traits to be all null, re-run sim → that character should appear in failures. Restore.

M2.7Question difficulty tiering
Tag questions easy vs nerdy; engine holds nerdy ones in reserve until it misses once or hits a turn threshold. Keeps casual fans in the game without dumbing it down for diehards.
▾

~3hUX

Why

The current bank asks things like “Did your character first appear during the Bronze Age (1970–1985)?” or “Does your character originate in Madripoor?” — correct for the trait, useless for a player who just knows Spider-Man from the movies. A casual fan currently gets surfaced these in the first few turns because the entropy selector doesn’t know they’re hard. We need a softer ramp.

Step-by-step

  1. Extend Question type in src/lib/engine/types.ts with difficulty: "easy" | "nerdy". Default to "easy" in the schema.
  2. Hand-tag the ~20–25 nerdy questions in data/questions.json: all 10 era questions, obscure origin-city (Madripoor, Themyscira-as-trivia, Krypton specifics), death-resurrection, and niche secret-identity. Keep universe/alignment/gender/visible appearance/MCU·DCEU adaptations as easy.
  3. Validate at build time: extend scripts/validate-dataset.ts to enforce that every question carries a valid difficulty and that at least 60% are easy.
  4. Update src/lib/engine/select.ts: filter the question pool to easy-only when state.wrongGuesses.length === 0 && state.turn < NERDY_UNLOCK_TURN (start at 12). Once unlocked, the full pool is in play.
  5. If the easy pool exhausts before unlock, allow nerdy questions early (don’t stall the engine).
  6. Re-run npm run sim; expect avg questions to rise slightly on diehard targets but drop on mainstream ones. Adjust the unlock turn against the report.

Manual Testing

  1. Start a game, play through the first 10 turns → every question is answerable from movie/show knowledge (no era / no obscure city / no death-resurrection deep cuts).
  2. Get to a wrong guess → from the next turn the question pool can include nerdy ones.
  3. Run npm run sim → aggregate accuracy is ±3% of the pre-tiering baseline; report saved.
  4. Pick a mainstream character (Spider-Man, Batman, Wonder Woman) → engine guesses in ≤ 15 turns without ever asking an era question.
  5. Pick an obscure character that needs nerdy traits to disambiguate → engine still reaches them; the nerdy question shows up only after the easy pool is mostly consumed or after a miss.
  6. npm run dataset:validate rejects a tampered questions.json with an invalid difficulty.

Done when: all 6 tests pass.
