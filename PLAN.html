M1.2Data pipeline script
Merge FiveThirtyEight CSVs + Superhero API JSON into a deduped characters.json. Disk-only, no live fetches.
▾

~3hdata risk

Goal

One offline script that produces a canonical data/characters.json from three pre-downloaded source files. Everything downstream — the engine, the UI, the sim — reads from this file. The pipeline never touches the network: all sources are downloaded once by the user (sidesteps corp-SSL inspection).

Sources (all free, no account, no key)

  • FiveThirtyEight Marvel CSV — raw.githubusercontent.com/.../marvel-wikia-data.csv → save to scripts/sources/marvel.csv. Provides identity, alignment, gender, eye/hair colour, first-appearance year, comic-appearance count.
  • FiveThirtyEight DC CSV — raw.githubusercontent.com/.../dc-wikia-data.csv → save to scripts/sources/dc.csv.
  • Superhero API JSON dump — akabab.github.io/superhero-api/api/all.json → save to scripts/sources/superhero-api.json. 731 characters with powerstats, biography, appearance, work, connections, and an embedded image.url. Images come from this source — no Wikipedia lookups, no extra HTTPS calls in the pipeline.

The original plan called for Kaggle + Wikipedia image lookups. Both required either an account or ~500 live HTTPS calls (which the corp SSL filter intercepts). FiveThirtyEight is the same comic-characters data Kaggle hosts but free on GitHub; Superhero API's embedded image URLs cover every shipped character for MVP. Phase 2 can swap in Wikipedia thumbnails if image URL rot becomes an issue.

Step-by-step

  1. Create scripts/sources/ and download the 3 files above into it (right-click → Save link as, or wget from a non-corp machine). Commit them — they’re part of the build, not artifacts.
  2. Install the one new dep: npm i -D csv-parse
  3. Build scripts/lib/normalize.ts with the canonical Character type and helpers: slugify(), dedupeKey(), normalizePublisher(), normalizeAlignment(), normalizeGender(), normalizeIdentity(), parseYear(), cleanString(), cleanNumber(), countTraits(). All sources funnel through these so the schema stays uniform.
  4. Build scripts/build-dataset.ts: (a) load Superhero API, filter to Marvel/DC entries with an image → (b) load FiveThirtyEight CSVs into lookup maps keyed by dedupeKey(name) → (c) for each Superhero entry, merge in matching FT row to fill appearances, yearFirstAppeared, identity, and any missing alignment/gender/eye/hair → (d) dedupe by id, keeping the richer entry → (e) score completeness = populated trait count / 15 → (f) select top 500 by completeness × (1 + log10(appearances + 1)) → (g) write to data/characters.json.
  5. Build scripts/validate-dataset.ts: assert array of ≥400 entries, every one has id + name + publisher + image + ≥8 non-null traits, no duplicate ids, publisher is Marvel or DC. Exit nonzero on violation (this gates npm run build).
  6. Run npm run dataset:build. Read the stdout report — min/avg/max completeness, Marvel/DC split, top 10 by appearances, any famous-hero misses.
  7. Commit data/characters.json (the source of truth for the app, not a build artifact).

Files touched

  • scripts/sources/marvel.csv, dc.csv, superhero-api.json (downloaded by user)
  • scripts/lib/normalize.ts — schema + helpers
  • scripts/build-dataset.ts, scripts/validate-dataset.ts
  • data/characters.json (generated, committed)

Testing

  1. Run: npm run dataset:build
  2. Look for: stdout reports loading Superhero API + both FiveThirtyEight CSVs, augmentation count, then Wrote ~500 characters to data/characters.json. No stack traces.
  3. Look for: the completeness summary line — avg should be ≥ 70%. If it’s much lower, the FiveThirtyEight merge is misaligned (check that you saved as marvel.csv / dc.csv, not e.g. marvel-wikia-data.csv).
  4. Look for: the top 10 by appearances list — should include Spider-Man, Batman, Superman, Iron Man, Wolverine, or similar. If you see deep cuts like “A-Bomb” or “Agent 13” topping the list, the merge with FiveThirtyEight failed.
  5. Look for: the heads-up: N canonical heroes not in final set line — should say “all canonical top heroes present” or list at most 1–2 misses.
  6. Run: npm run dataset:validate
  7. Look for: exit 0 with validate-dataset: N characters OK. Nonzero with a numbered list of violations means the build is shipping bad data.
  8. Run: open data/characters.json in your editor; search for “Spider-Man”, “Batman”, “Wonder Woman”.
  9. Look for: each entry has a populated image field (URL), at least 8 non-null trait fields, sane publisher + alignment.
  10. Run: pick 5 random image URLs from the JSON and paste them into your browser.
  11. Look for: the image loads. If 1–2 are broken (URL rot at the source), tolerable for MVP. If most are broken, the Superhero API JSON is stale — redownload.
  12. Red flag: fewer than 400 entries, validation exits nonzero, top-10-by-appearances is all unknowns, or completeness avg below 50%.

Done when:

  • data/characters.json exists with ~500 entries.
  • Every entry has a populated image URL and ≥8 non-null traits.
  • npm run dataset:validate exits zero.
  • Top-10-by-appearances list contains recognizable mainstream Marvel/DC heroes.
M1.3Trait taxonomy & question bank
~120 traits across 14 categories. ~150 questions with inverse phrasing.
▾

~5hcreative

Goal

A stable controlled vocabulary of traits + a hand-authored set of questions that map to them. The engine's question-picker can only ask what is in this file.

Step-by-step

  1. Create data/traits.json as { id, category, label, polarity }. Categories (from your questionnaire choice): Universe, Alignment, Gender, Species, Powers, Equipment, Teams, Era, Appearance, Costume, Secret Identity, Origin City, Adaptations, Death/Resurrection. Aim for ~120 trait IDs total.
  2. Backfill trait values into characters.json — for each trait, set boolean or 0–1 weight where known, null where unknown. For MVP, focus on the top 500 and accept many nulls in lesser-known characters — the engine handles unknowns.
  3. Create data/questions.json as { id, trait, text, inverse, category, cost: 1 }. ~150 entries. Examples: "Does your character have superhuman strength?", inverse "No super strength". Aim for 2–3 questions per high-signal trait so the engine has variety.
  4. Add a sanity script in validate-dataset.ts: every question's trait must exist in traits.json; every trait must appear in ≥ 1 question.
  5. Manual sweep: read every question aloud — if it could be misread or has cultural ambiguity, rewrite.

Done when:

  • traits.json has ~120 entries covering all 14 categories.
  • questions.json has ~150 entries, all mapping to real traits.
  • Validation passes.
  • Each top-50 character has ≥ 15 non-null trait values.
M1.4Entropy engine module
Bayesian posterior + info-gain question selection + adaptive guess + wrong-guess loop.
▾

~10hcore IP

Goal

The brain. A pure-TypeScript module with no React, no DOM, no I/O. Inputs: dataset + current answer history. Outputs: next question or a guess.

Module layout (src/lib/engine/)

  • types.ts — Answer, Belief, GameState, EngineDecision
  • likelihood.ts — maps (answer, traitValue) → multiplier (see plan’s answer table)
  • posterior.ts — update(belief, question, answer, chars) with min-floor 1e-6, renormalize
  • entropy.ts — entropy(belief)
  • select.ts — nextQuestion(state, pool) with info-gain + diversity penalty (0.85x if category is in last 2)
  • guess.ts — shouldGuess(belief, turn) + shouldGiveUp(belief, turn, missCount)
  • index.ts — the high-level decide(state) orchestrator

Step-by-step

  1. Write types.ts first — pin the shapes before any logic.
  2. Implement likelihood.ts. Unit-test the 5 answer types against trait values 0, 0.5, 1, and null.
  3. Implement posterior.ts. Unit-test: a "Yes" to a trait where only one character has value 1.0 should drive that character’s posterior to ~1.0 after one update.
  4. Implement entropy.ts (Shannon, base 2). Unit-test entropy(uniform) ≈ log2(N).
  5. Implement select.ts. Test: with a clear binary split (half characters Marvel, half DC), "Is your character from Marvel?" wins expected info-gain.
  6. Implement guess.ts with the threshold table from the plan (turn ≥ 8 + top1 > 0.85 or top1/top2 > 6, soft cap 20, hard cap 25).
  7. Implement index.ts — returns { kind: 'question', q } | { kind: 'guess', characterId } | { kind: 'giveup' }.
  8. Cover everything with Vitest. Target 90%+ coverage on the engine folder; it's the riskiest code.

Done when:

  • All engine unit tests pass.
  • Engine is pure: no window, no localStorage, no React imports.
  • Manually scripting a game with 15 Yes/No answers against a known character lands on the right guess in <25 turns.
M1.5Simulation harness baseline
Auto-play every character. Report accuracy, avg questions, worst offenders.
▾

~3himportant

Goal

An offline benchmark that plays a full game for every character in the dataset, using that character’s own traits as the oracle. Gives you a single number for "how good is the engine right now?" and a list of characters the engine struggles with — gold for tuning.

Step-by-step

  1. Create scripts/sim.ts.
  2. For each character C in the dataset: initialize a fresh GameState; loop calling engine.decide(state); when it asks a question, answer using C’s trait value (1.0→Yes, 0.0→No, 0.4–0.6→Probably, null→Don’t know); when it guesses, record hit/miss; if miss, continue per the wrong-guess loop; cap at 30 turns.
  3. Aggregate: % correctly guessed, average question count, P95 question count, top 20 characters by failure.
  4. Write a CLI flag --noise=0.1 that flips 10% of oracle answers to "Don’t know" to simulate fuzzy human players.
  5. Save the report to sim-report.json for diff-friendly tracking across commits.

Done when:

  • npm run sim finishes in under 60s for 500 characters.
  • Report shows a baseline accuracy number (any number — tuning happens in M4.1).
  • Top-20-failures list is human-readable; you can eyeball why the engine missed (sparse traits, ambiguous answers, etc.).

~6h (iterative)tuning

Step-by-step

  1. Run npm run sim — check baseline from M1.5.
  2. Look at the top-20 failures. Are they failing because of (a) sparse trait data, (b) ambiguous questions, or (c) the engine not driving enough info-gain from available traits?

  6. Iterate. Commit a sim report alongside any tuning change so you can see the curve.
  7. Add a CI step (GitHub Actions) running npm run sim -- --ci that exits nonzero if accuracy regresses by >3% from sim-report.json on main.

P2.1Expand dataset to 3000+ characters
The headline Phase 2 goal — enrich the long tail without hurting engine accuracy.
▾

~20himportant

Strategy

The MVP shipped 500 well-traited characters. Adding 2500 more with sparse traits will hurt engine accuracy unless we’re careful. The plan is to enrich in waves and gate each by simulation.

Step-by-step

  1. Wave A — tier 2 popularity (next 500): Extend build-dataset.ts to take the next-most-popular 500 characters from Superhero API + Kaggle. Run sim — expect a 5–10pt accuracy drop.
  2. Trait backfill from Fandom infoboxes: Build scripts/enrich-from-fandom.ts. For each character with < 12 known traits, fetch their Fandom page, parse the infobox HTML, map fields to your traits.
  3. Wave B — long tail (2000 more): Add the remainder. Mark them as tier: 'deep_cut'.
  4. Difficulty toggle on landing: "Just the famous ones" (tier 1, default) vs "Include deep cuts" (all tiers). Filter the engine’s pool at game start.
  5. Re-tune costs: Some questions that worked at 500 chars will be weaker at 3000 — re-run M4.1’s tuning loop.
  6. Image pipeline cost: 3000 hot-linked images is fine; Wikipedia handles it. If Fandom rate-limits, fall back to silhouettes.

Done when:

  • 3000+ characters live in characters.json.
  • Sim: ≥85% on tier 1 (unchanged from MVP), ≥65% on full pool.
  • Difficulty toggle works.
  • Bundle size still acceptable (move dataset to a separate /data fetch if it grows past 1 MB).
